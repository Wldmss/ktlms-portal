## 3. `joda-time` -> `java.time`

컴파일은 되나 스프링 6 / MyBatis 최신 버전과 날짜 데이터 바인딩 시 한글 깨짐, DB 서식 오류, 날짜 유실 발생

| 기능              | Joda-Time (구형 ❌)                      | Java.Time (신형 ⭕)                         | 비고                 |
|-----------------|---------------------------------------|------------------------------------------|--------------------|
| **날짜+시간**       | `org.joda.time.DateTime`              | **`java.time.ZonedDateTime`**            | 타임존이 포함된 날짜/시간 데이터 |
| **로컬 날짜+시간**    | `org.joda.time.LocalDateTime`         | **`java.time.LocalDateTime`**            | 타임존이 없는 날짜/시간 데이터  |
| **날짜만**         | `org.joda.time.LocalDate`             | **`java.time.LocalDate`**                | 년-월-일 데이터          |
| **시간만**         | `org.joda.time.LocalTime`             | **`java.time.LocalTime`**                | 시:분:초 데이터          |
| **절대 시간 (초)**   | `org.joda.time.Instant`               | **`java.time.Instant`**                  | 타임스탬프 처리용          |
| **시간차 (밀리초)**   | `org.joda.time.Duration`              | **`java.time.Duration`**                 | 시/분/초 단위의 간격 측정    |
| **날짜차 (년/월/일)** | `org.joda.time.Period`                | **`java.time.Period`**                   | 일/월/년 단위의 간격 측정    |
| **포맷 변환기**      | `org.joda.time.format.DateTimeFormat` | **`java.time.format.DateTimeFormatter`** | 문자열 ↔ 날짜 변환기       |

## 🟠 jdk, spring 등 관련 라이브러리 버전만 변경 시

* 최소한의 변경, 기존 프로젝트 유지 방법
* build.xml 에 있는 jar 파일들의 전체 버전 변경 필요
    * jar 파일 다운로드 및 변경 작업 필요
    * javax 스펙이 포함된 jar 를 모두 찾아 jakarta 관련 버전으로 업그레이드 필요
* web.xml 스펙 선언문 6.0 업그레이드
* servlet-context.xml(KT-*.xml) 6.0 규격으로 변경
* 미사용 또는 변경된 라이브러리 관련된 설정 파일, 라이브러리 등 삭제 필요
* jackson 버전 변경 필요 (2.9.0 -> 2.18.6)

### 예상 이슈

* 라이브러리 버전이 꼬이는 경우, 라이브러리 버전 맞추는데 시간이 소요됨
* 추후 유지보수 등 기존과 동일한 공수 예상
    * 근본적인 원인 해결이 아니기 때문
    * 보안 이슈 발생 시 jar 파일을 매번 변경해줘야 함
* git action 사용 시 프로젝트 내에 jar 파일이 포함되어 있기 때문에 무거워짐 -> ci script 에 ant 강제 설정 필요
* 당장 이번 애저전환 시에는 시간이 적게 들고, 수정사항이 적겠지만, 유지보수를 진행하다보면 더 많은 시간이 듦

## 🟠 ant 형식(build.xml-jar 파일 수동 import) -> maven 변경

* nexus 를 사용하기 위해 maven 으로 변경에 따른 프로젝트 전체적인 구조 변경 필요
* 환경 및 구조 설정 완료 후 사용하는 페이지만 지정된 폴더로 이전하면서 오류 수정
* /conf/spring 하위 KT-*.xml 통합 필요 (servlet-context.xml, root-context.xml)

### 예상 이슈

* 처음에는 구조 잡고 오류 개선, 누락된 라이브러리 추가 등 작업이 필요하지만 어느정도 진행되다보면 동일한 패턴으로 작업 가능
* xml 설정 파일, 환경 설정 파일이 여기저기 분리되어 있어 통합 시 누락되는 부분 발생 가능 -> 누락이 없는지 크로스 체크 필요

## 🟢 기타 고려 사항

### ✅ 로그인 방식

* 기존 CommonMap 에 사용자 데이터를 넣어두고 사용하는 방식 -> jwt token 처리
* 보안성 검토 시 이 부분에 사번, 이름, 조직코드 등 이슈가 있어 임시 처리해두었던 부분들 전체적인 검수 필요

### ✅ gnb, footer (GNBLayout 등) 모든 페이지 공통 부분 자동 주입 처리

* 메뉴별로 옮길 때 불필요한 중복 부분만 잘 삭제하고, 파일 구조만 지키면 앞으로가 편함

### ✅ css 모듈화

* 기존 new23.css, common.css 등등 애매하게 뭉쳐있는 css 들을 공통 / 기능별 / 메뉴별 파일 분리
* 사용하지 않는 css, css 파일 삭제

### ✅ 환경 변수 통합

* spring(boot) 표준 파일명인 `application.properties` 로 환경 변수 파일 통합
* 공통, 로컬(local), 개발(dev), 운영(prod) 로 프로파일별 변수 관리

### ✅ 공통 기능 모듈화

* alert, modal, snackbar, loading bar 등 공통 모듈로 전환

### ✅ tomcat -> jboss wildfly

* 로컬 환경에서도 jboss 로 서버 실행

### ✅ js 라이브러리 webjars 사용

* js 라이브러리를 기존 min.js, min.css 파일을 로드하는 방식 -> 주요 라이브러리만 webjars maven 으로 불러오는 방식 사용
* nexus 에 업로드 요청 필요 (jquery 는 있음)
* webjars 로 불러오게 되면 jar 로 받아오기 때문에 조금 더 무거워짐
* jquery, bootstrap 만 webjar 사용, 나머지는 기존처럼 min.js, min.css 라이브러리 /resources/vendor 하위에 폴더로 관리
